---
title: "File tools: read, write, edit, glob, and grep"
description: "Reference for Claw Code's five file tools: ReadFile, WriteFile, EditFile, GlobSearch, and GrepSearch. Covers parameters, size limits, and security guards."
---

The five file tools cover every common file operation inside the workspace. Four of them require only `read-only` permission; writes and edits require `workspace-write`. All five enforce a hard workspace boundary — paths that escape the workspace root via `../` traversal or symlink following are rejected before any I/O occurs.

File-tool behavior is implemented in `rust/crates/runtime/src/file_ops.rs` (744 LOC). Edge-case guards — binary detection, size limits, workspace boundary validation, and symlink escape prevention — were added in lane 3 of the 9-lane checkpoint (feature commit `284163b`).

<AccordionGroup>

  <Accordion title="read_file">

    Reads a text file from the workspace and returns its contents. Binary files are rejected via NUL-byte detection before any content is returned.

    **Permission:** `read-only`

    **Size limit:** `MAX_READ_SIZE` — reads larger than this limit are truncated or rejected.

    <ParamField path="path" type="string" required>
      Workspace-relative or absolute path to the file. Must resolve inside the workspace root.
    </ParamField>

    <ParamField path="offset" type="integer">
      Line number to start reading from (0-indexed). Use with `limit` for paginated reads of large files.
    </ParamField>

    <ParamField path="limit" type="integer">
      Maximum number of lines to return. Defaults to the full file up to `MAX_READ_SIZE`.
    </ParamField>

    **Security guards:**
    - Workspace boundary check: canonical path must be a descendant of the workspace root.
    - Symlink escape prevention: symlinks are resolved before the boundary check.
    - Binary file detection: files containing NUL bytes are rejected with a descriptive error.

    **Parity harness coverage:** `read_file_roundtrip` — verifies read-path execution and final synthesis in the mock parity harness.

    ```json
    {
      "path": "src/main.rs",
      "offset": 0,
      "limit": 100
    }
    ```

  </Accordion>

  <Accordion title="write_file">

    Creates or overwrites a text file in the workspace.

    **Permission:** `workspace-write`

    **Size limit:** `MAX_WRITE_SIZE` — content larger than this limit is rejected.

    <ParamField path="path" type="string" required>
      Workspace-relative or absolute path for the target file.
    </ParamField>

    <ParamField path="content" type="string" required>
      Full text content to write. Must not exceed `MAX_WRITE_SIZE`.
    </ParamField>

    **Security guards:**
    - Workspace boundary enforcement: the target path must be inside the workspace root. Writes outside the boundary return a `workspace-boundary` denial.
    - Read-only mode blocks all writes. The `PermissionEnforcer::check_file_write()` method evaluates the current `PermissionMode` and rejects writes in `read-only` sessions before any I/O occurs.
    - Symlink escape prevention applies the same canonical-path check used by `read_file`.

    **Parity harness coverage:**
    - `write_file_allowed` — validates a successful write in `workspace-write` mode.
    - `write_file_denied` — validates that a write is rejected in `read-only` mode with a structured denial payload.

    ```json
    {
      "path": "output/report.md",
      "content": "# Report\n\nGenerated by Claw Code."
    }
    ```

  </Accordion>

  <Accordion title="edit_file">

    Replaces a specific string in an existing workspace file. Useful for targeted edits without rewriting the entire file.

    **Permission:** `workspace-write`

    <ParamField path="path" type="string" required>
      Path to the file to modify.
    </ParamField>

    <ParamField path="old_string" type="string" required>
      The exact string to search for in the file. The edit fails if the string is not found.
    </ParamField>

    <ParamField path="new_string" type="string" required>
      The replacement string. May be empty to delete the matched text.
    </ParamField>

    <ParamField path="replace_all" type="boolean">
      When `true`, replaces every occurrence of `old_string`. When `false` (default), only the first occurrence is replaced.
    </ParamField>

    **Security guards:** Same workspace boundary and symlink escape checks as `write_file`. The permission enforcer applies `WorkspaceWrite` mode gating before any file mutation.

    ```json
    {
      "path": "src/config.rs",
      "old_string": "const VERSION: &str = \"1.0.0\";",
      "new_string": "const VERSION: &str = \"1.1.0\";",
      "replace_all": false
    }
    ```

  </Accordion>

  <Accordion title="glob_search">

    Finds files matching a glob pattern, scoped to an optional subdirectory.

    **Permission:** `read-only`

    <ParamField path="pattern" type="string" required>
      Glob pattern to match. Examples: `**/*.rs`, `src/**/*.toml`, `*.md`.
    </ParamField>

    <ParamField path="path" type="string">
      Optional root directory to search from. Defaults to the workspace root.
    </ParamField>

    Returns a list of matching file paths sorted by modification time.

    ```json
    {
      "pattern": "**/*.rs",
      "path": "rust/crates/tools"
    }
    ```

  </Accordion>

  <Accordion title="grep_search">

    Searches file contents using a regular expression pattern. Supports context lines, case-insensitive matching, multiline mode, and file-type filtering.

    **Permission:** `read-only`

    <ParamField path="pattern" type="string" required>
      Regular expression to match against file contents. Full regex syntax is supported.
    </ParamField>

    <ParamField path="path" type="string">
      Directory to search. Defaults to the workspace root.
    </ParamField>

    <ParamField path="glob" type="string">
      Restrict the search to files matching this glob pattern (e.g. `*.rs`).
    </ParamField>

    <ParamField path="output_mode" type="string">
      Controls result verbosity. Defaults to match lines with file/line numbers.
    </ParamField>

    <ParamField path="-B" type="integer">
      Lines of context to show before each match.
    </ParamField>

    <ParamField path="-A" type="integer">
      Lines of context to show after each match.
    </ParamField>

    <ParamField path="-C" type="integer">
      Symmetric context lines before and after each match (shorthand for equal `-B`/`-A`).
    </ParamField>

    <ParamField path="-i" type="boolean">
      Enable case-insensitive matching.
    </ParamField>

    <ParamField path="-n" type="boolean">
      Include line numbers in output.
    </ParamField>

    <ParamField path="multiline" type="boolean">
      Enable multiline regex mode (`.` matches newlines).
    </ParamField>

    <ParamField path="head_limit" type="integer">
      Maximum number of matching lines to return.
    </ParamField>

    <ParamField path="offset" type="integer">
      Skip the first N matches (for pagination).
    </ParamField>

    <ParamField path="type" type="string">
      Restrict to a named file type (e.g. `rust`, `json`).
    </ParamField>

    **Parity harness coverage:** `grep_chunk_assembly` — validates chunked grep tool output handling and chunk reassembly in the mock parity harness.

    ```json
    {
      "pattern": "fn run_",
      "path": "rust/crates/tools/src",
      "glob": "*.rs",
      "-n": true
    }
    ```

  </Accordion>

</AccordionGroup>

## Using `--allowedTools` to restrict file tools

When running in automation contexts, you can expose only file tools to limit the model's surface:

```bash
./target/debug/claw --allowedTools read_file,glob_search,grep_search \
  --permission-mode read-only \
  prompt "summarize all Rust source files"
```

Short aliases are accepted: `read`, `write`, `edit`, `glob`, `grep`.
